# Access Control Policy — Malon

**Version:** 1.0
**Last Updated:** 2026-07-16
**Status:** Active
**Classification:** Internal — Enterprise

---

## 1. Purpose and Scope

This policy governs access to the Malon MCP server, its administrative interfaces,
API keys, secret material, and the codebase that implements it. It applies to all
human engineers, automated agents, CI/CD pipelines, and third-party integrations
that interact with the Malon repository, build pipeline, npm publishing, or hosted
services.

**Scope boundaries:**

- Source code repository (GitHub: `main` branch and all forks)
- npm package publishing (Trusted Publishing via OIDC)
- CI/CD pipeline (GitHub Actions)
- Memory ledger contents (`.malon/memory/`)
- Auth subsystem (API keys, sessions, MFA tokens)
- Configuration (`.malon/config.yml`, environment variables)

## 2. Roles and Responsibilities

### 2.1 Role Definitions

Malon implements a **Role-Based Access Control (RBAC)** system with five tiers,
enforced programmatically in `src/auth/rbac.ts`:

| Role         | Permissions                                                                 | Typical Holder         |
| ------------ | --------------------------------------------------------------------------- | ---------------------- |
| **admin**    | Full system access: generate/revoke keys, modify any config, all operations | Founder, Security Lead |
| **operator** | Session management, status monitoring, checkpoint control                   | DevOps, SRE            |
| **service**  | Automated tool calls (search, memory read/write) via API keys               | CI/CD, Agent instances |
| **user**     | Read-only search, memory access, status                                     | Individual developers  |
| **viewer**   | Status and health checks only                                               | Monitoring, dashboards |

### 2.2 Responsibility Matrix

| Activity                    | Admin | Operator | Service | User | Viewer |
| --------------------------- | ----- | -------- | ------- | ---- | ------ |
| Generate API keys           | ✓     |          |         |      |        |
| Revoke API keys             | ✓     |          |         |      |        |
| List API keys               | ✓     | ✓        |         |      |        |
| Configure MFA               | ✓     | ✓        |         |      |        |
| Run `malon_search`          | ✓     | ✓        | ✓       | ✓    |        |
| Run `malon_memory_write`    | ✓     | ✓        | ✓       |      |        |
| Run `malon_memory_get`      | ✓     | ✓        | ✓       | ✓    |        |
| Run `malon_checkpoint`      | ✓     | ✓        | ✓       | ✓    |        |
| Run `malon_status`          | ✓     | ✓        | ✓       | ✓    | ✓      |
| Modify server config        | ✓     |          |         |      |        |
| View audit / usage logs     | ✓     | ✓        |         |      |        |
| Trigger full re-index       | ✓     | ✓        |         |      |        |
| Publish npm release         | ✓     |          |         |      |        |
| Merge PRs to `main`         | ✓     |          |         |      |        |
| Merge PRs to release branch | ✓     |          |         |      |        |

### 2.3 Separation of Duties

The following pairs of activities **must not** be performed by the same entity:

1. **npm publish** and **code merge to release branch** — The release branch PR
   requires a separate human approval beyond the standard PR review (per §2.7 of AGENTS.md).
2. **Security test authorship** and **code merge** — Security tests are written
   by the agent; the founder approves the merge after reading the pass/fail summary.
3. **API key generation** and **API key usage** — Keys are generated by an admin
   and consumed by services/users. The key hash is stored; the raw key is shown
   exactly once at creation.

## 3. Authentication Mechanisms

### 3.1 API Keys

| Property     | Specification                                                |
| ------------ | ------------------------------------------------------------ |
| Prefix       | `mal_`                                                       |
| Length       | 32 bytes (44 base64url chars)                                |
| Storage      | SHA-256 hash only; raw key shown once at creation            |
| Comparison   | Timing-safe (`crypto.timingSafeEqual`)                       |
| Revocation   | Immediate via `malon_admin` tool with `revoke-key` operation |
| Max per user | Configurable in `auth.max_api_keys_per_user` (default: 10)   |

Implementation: `src/auth/api-key.ts`

### 3.2 Session Tokens

| Property | Specification                                     |
| -------- | ------------------------------------------------- |
| Length   | 48 bytes random                                   |
| Encoding | Base64url                                         |
| Storage  | SHA-256 hash only                                 |
| Expiry   | Configurable TTL (default: 24 hours)              |
| Refresh  | Allowed within TTL; extends TTL from refresh time |

Implementation: `src/auth/session.ts`

### 3.3 Multi-Factor Authentication (MFA)

| Property       | Specification                                         |
| -------------- | ----------------------------------------------------- |
| Type           | TOTP (Time-based One-Time Password)                   |
| Algorithm      | SHA-1, 30-second window                               |
| Skew tolerance | +/-1 window (3 attempts total per 30s)                |
| Recovery codes | 8 single-use codes generated at setup                 |
| Enforced       | Configurable via `auth.mfa_enforced` (default: false) |

Implementation: `src/auth/mfa.ts`

## 4. Access Review and Certification

### 4.1 Review Cadence

| Review Type           | Frequency | Performed By | Artifact                                    |
| --------------------- | --------- | ------------ | ------------------------------------------- |
| API key audit         | Monthly   | Admin        | List all active keys, verify purpose labels |
| Session audit         | Weekly    | Automated    | Prune expired sessions; log stale sessions  |
| RBAC role review      | Quarterly | Founder      | Confirm role assignments match current team |
| Access policy review  | Annually  | Founder      | Update this document; re-approve changes    |
| MFA enforcement check | Quarterly | Admin        | Verify all admin/operator accounts have MFA |

### 4.2 Review Procedure

1. **List active principals:** `malon_admin` with `list-keys` operation
2. **Verify each key label:** Contact the key owner (if known) or revoke orphaned keys
3. **Check creation dates:** Revoke keys older than 90 days without recent use
4. **Document findings:** Append review results to `.malon/memory/decisions.md`
5. **Remediate:** Revoke flagged keys; notify affected users

## 5. Provisioning and Deprovisioning

### 5.1 Access Grant Process

```
Request → Authorization → Provisioning → Notification → Verification
   │           │              │               │              │
   ▼           ▼              ▼               ▼              ▼
 Ticket    Admin          Generate       Notify         Test
 filed    approves       API key /      owner via     access
 by       (in ticket)   configure       existing      works
 manager                 session        channel       as expected
```

### 5.2 Access Revocation Process

```
Trigger → Identification → Revocation → Confirmation → Logging
   │           │              │               │            │
   ▼           ▼              ▼               ▼            ▼
 Employee   Admin runs    Key revoked     Verify key    Log entry
 leaves /  `malon_admin`  immediately    no longer     in usage
 role       revoke-key    (hash removed  works (401)    log
 changes                  from store)
```

**Revocation SLA:**

- Employee departure: within 4 hours of notification
- Role change: within 24 hours
- Suspected compromise: immediate (P0 incident per IR plan)

## 6. Principle of Least Privilege

1. **Default-deny:** All permissions are denied unless explicitly granted by role.
   Enforced in `src/auth/rbac.ts:hasPermission()`.
2. **Time-bound grants:** API keys and sessions have configurable TTLs. No
   permanent credentials.
3. **Scope isolation:** API keys are scoped to a specific repo instance. A key
   generated for one repo cannot be used against another.
4. **Tool-level boundaries:** Each MCP tool is structurally incapable of doing
   more than its stated job (§2.5 of AGENTS.md). `malon_search` is read-only;
   `malon_memory_write` is scoped to `.malon/memory/`.

## 7. Audit Logging

Every access-control event is logged to `usage.log` with:

| Field        | Description                                          |
| ------------ | ---------------------------------------------------- |
| `timestamp`  | ISO 8601 UTC                                         |
| `session_id` | UUID v7                                              |
| `event`      | `key_generated`, `key_revoked`, `auth_failure`, etc. |
| `principal`  | Key ID prefix or session token hash prefix           |
| `action`     | The operation attempted                              |
| `allowed`    | Boolean — was the access granted?                    |
| `reason`     | If denied: the specific permission that was missing  |

Logs are append-only, retained per `retention.usage_log_max_age_days` (default: 30 days),
and are gitignored. No secrets or full tokens appear in any log line.

## 8. Exceptions and Escalation

| Situation                               | Process                                                                     |
| --------------------------------------- | --------------------------------------------------------------------------- |
| Emergency access needed outside policy  | Founder approves via out-of-band channel (phone/Signal); document within 1h |
| Role assignment needs temporary upgrade | Founder approves; auto-reverts after 24 hours unless renewed                |
| Policy conflict with customer contract  | Customer contract takes precedence; document exception in ADR               |
| Automated agent needs elevated access   | Generate scoped service key with minimum required role; rotate quarterly    |

---

## 9. Enforcement

This policy is enforced at four layers:

1. **Code-level** — `src/auth/rbac.ts`, `src/auth/api-key.ts`, `src/auth/session.ts`
   enforce RBAC, key validation, and session expiry programmatically.
2. **Test-level** — `test/security/auth-security.test.ts` (30+ tests) asserts
   key format, hash/verify, TOTP windows, RBAC permission matrix, collision entropy.
3. **CI-level** — Security tests are a release gate. PRs cannot merge if auth
   tests fail.
4. **Repo-level** — Branch protection on `main` requires PR + review. No
   direct pushes. Release branch requires separate approval.

## 10. Policy Violations

Violations of this policy are security incidents and follow the
Incident Response Plan (`docs/INCIDENT_RESPONSE_PLAN.md`). Severity
classification:

| Violation                                    | Severity |
| -------------------------------------------- | -------- |
| Sharing API keys outside authorized channels | P1       |
| Using elevated role without authorization    | P1       |
| Failing to revoke access within SLA          | P2       |
| Skipping access review                       | P3       |
| MFA bypass / sharing recovery codes          | P1       |
